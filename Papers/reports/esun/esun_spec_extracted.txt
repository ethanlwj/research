for network controllers and switches that support scale-up GPU connectivity, clearly outlining
the features necessary for at-scale deployment in AI infrastructure. By publishing these

This specification represents an initial version that establishes a functional baseline

of lossless, differentiated, and resilient scale-up networks.
2.2. Efficiency
The OCP specification for ESUN (Ethernet for Scale Up Networks) advances efficiency by
standardizing scale-up GPU connectivity and hierarchical network topologies. By defining
ESUN header requirements, congestion management, and differentiated traffic handling, it
eliminates costly custom solutions and streamlines deployment. Optimized packet headers
reduce protocol overhead, improving goodput and lowering latency, which maximizes
compute utilization and reduces energy consumption. Enhanced congestion management
and load balancing further boost operational efficiency, minimizing idle time for expensive
GPU resources.
The specification also preserves existing hardware and software investments by maintaining
backward compatibility and providing a consistent framework for resilient, differentiated
networking. This approach reduces adaptation costs and avoids unnecessary duplication of
effort, directly supporting the Open Compute Project’s tenet of efficiency by enabling
sustainable, cost-effective growth in AI infrastructure.

3. Scope
This document provides Base Specification for ESUN.
OCP ESUN (Ethernet for Scale Up Networks) is about evolving Ethernet to meet the
unique demands of Scale-Up AI infrastructure. ESUN (used interchangeably as “OCP
ESUN”) builds on Ethernet by introducing evolutionary enhancements that improve
goodput, latency, and reliability to meet the demands of Scale-Up GPU connectivity.
4. Trends Driving Scale-Up Networks
Scale-up networks for GPUs are high-performance interconnects designed to facilitate
efficient, low-latency communication among multiple GPUs within a tightly coupled
cluster. The primary traffic on Scale-up networks is Model Parallel Traffic which demands
High Bandwidth and Low Latency.
4.1. Low Latency:
The scale-up network enables the model developer to treat all the GPUs in the domain
as a single large GPU, resulting in improved developer efficiency. Consequently, this
makes it very important to have a stable network to ensure GPUs in the domain are not
stalled. Increasing the size of the scale-up networks gives a larger memory domain for
the model developers to work with, which in turn lowers communication latency and
provides end-to-end training and distributed inference speedups. Essentially, the
application now is trending towards smaller message sizes and requires lower latency
across a larger domain. This is critical for the success of future Models such as LLAMA,
GPT-OSS, DeepSeek, etc.
4.2. High Performance:
Our goal is to reduce compute Idle time for GPUs (an expensive resource). A highly
performant scale-up network will allow us to achieve this goal. Optimal compute/network
interleaving can achieve near-peak performance. This drives the exploration of fused
kernels, integrating data movement control and compute-network interleaving for lower
latencies, increasing the need for more performant scale-up networks. A lower network
overhead can lead to increased goodput.

4.3. Stability
Scaling up these networks is challenging. Until Llama4, scale-up was 8 GPU’s contained
within a node. Next-gen platforms feature dedicated scale-up networks, expected to
grow to >=1K GPUs, expanding beyond a single "box" across racks.
Stability and scalability issues with increased GPU cluster size can impact training
uptime. A 20% bandwidth drop in a small 128-accelerator domain means 5% e2e
performance loss and higher operational cost/year.
5. Current Ethernet vs. ESUN
5.1. Header Efficiency
5.1.1. Current Ethernet Header
The following header demonstrates use of standard IPv4 or IPv6 based protocol
on standard Ethernet.
5.1.2. ESUN Header (EH)
ESUN optimizes the packet headers for smaller topologies by removing IP
headers and using L2 addresses for forwarding. Functionality that is lost due to
this removal of IP headers (E.g. ECN, DSCP and higher entropy due to IP
headers and sockets) is reintroduced through a new smaller header called
“ESUN Header” or EH.

Goodput efficiency for all the message sizes with this change.

5.3. Lossless, differentiated and resilient Hierarchical Topologies
Radix limitation of switches as well as number of lanes per GPU will lead to deployment
of hierarchical topologies for SU networks. Hierarchical topologies demand support for
transient as well as persistent congestion control to avoid packet loss and high latencies.
Resilience is required in these networks to deal with link level and path level failures.
And congestion avoidance is achieved through load balancing techniques (e.g. ECMP,
Dynamic Load Balancing).

5.3.2. ESUN Enhanced capabilities
ESUN proposes use of a feature, Link Level Retry, defined by Ultra Ethernet
Consortium (UECTM) to achieve link resiliency against link level errors. (This
feature can be helpful to make high speed links resilient against packet drops
due to high BER). The goal for LLR is to improve the quality of the channel by
reducing exposure to BER. Packet drops outside of the BER-induced errors will
still need to be handled by end-to-end recovery mechanisms.
ESUN also recommends an additional link level congestion control mechanism
for Credit Based Flow Control (CBFC), defined by UECTM
, that can provide more
memory efficient designs for the ESUN switch devices.

5.4. Security
Security is an important consideration. As ESUN header replaces IP header, it is no
longer compatible with IPSec.
MACSec can be used alongside ESUN to add confidentiality, integrity and authentication
at Ethernet Layer 2 level. This provides point-to-point link level encryption.

6. Product Requirements
6.1. End Point changes for ESUN
1. Support for PFC. Optional support for LLR/CBFC
a. Enable configurability of LLR and CBFC per port if supported.
2. ESUN Header support
a. Specifying CoS, Flow label, TTL
b. Responding to ECN and announcing capability
6.2. Switch Side changes for ESUN
1. Shall support PFC, CBFC and LLR
a. Shall enable configurability of LLR and CBFC per port.
2. ESUN Header support
a. Ethernet DA based (48b) forwarding
i. This is assuming that there is no IP header, and switches shall
forward packets based on Ethernet Destination Addresses in the
ESUN header that are configured statically in the switches (there
is no learning or aging of MAC addresses).
ii. Switches should not broadcast ESUN packets with unknown
addresses.
b. CoS mapping to TC:
i. In the presence of CoS in 802.1Q tag (if enabled by some
implementation) or CoS in CBFC and EH: EH-COS should take
precedence.
c. Flow label should be used for hashing in load balancing decisions if F bit
is set.
d. ESUN switches should be standard ECN capable and should indicate
congestion via ECN bits when appropriate.
e. TTL field shall be decremented during forwarding and shall be used for
removing loops.
