At a Glance

Agentic LLM Inference Storage Bottleneck Dual-path Scheduling Bandwidth Efficiency

Core Problem

In long-running agentic workloads, repeatedly loading weights from storage can dominate latency and throughput, especially when one request is split into many independent “turns.”

Main Idea

DualPath separates execution into two coupled paths: one for hot-prefill and one for speculative decoding, minimizing repeated storage reads while preserving correctness.

Why It Matters

This design targets the storage bandwidth wall directly and enables better scaling for agentic systems where model state reuse is critical.

Abstract (Plain-English)

The paper argues that as LLM agents perform long, multi-step workflows, storage movement (not just compute) becomes a primary bottleneck. The authors propose DualPath, a serving architecture that orchestrates a hot-prefill path and a speculative decoding path to reduce redundant model weight transfers and improve end-to-end serving efficiency. Experiments show substantial throughput and latency gains compared with common baselines.

Introduction: What Changes in Agentic Inference

  • Agentic inference has frequent pauses, tool calls, branching, and resumed generations.
  • These patterns create many short generation bursts that repeatedly trigger memory/storage activity.
  • The paper frames this as a storage-bandwidth mismatch: compute can be available while data movement throttles execution.
  • The proposed system is designed around this reality rather than assuming a single uninterrupted generation stream.

Main Contributions

1. New Bottleneck Framing

Shows that storage bandwidth can be the limiting resource in agentic LLM serving, even when raw accelerator compute is high.

2. DualPath Architecture

Introduces a dual-path execution design to handle hot-prefill and decoding with reduced read amplification.

3. Scheduling + System Optimizations

Combines scheduling logic and implementation choices to improve overlap and reduce repeated model fetches.

4. Empirical Validation

Demonstrates improvements across throughput, latency, and storage efficiency against prior serving stacks.

Key Challenges Identified

  • Read amplification: repeated weight movement for fragmented request lifecycles.
  • Turn-level fragmentation: many short decode episodes break cache locality.
  • Prefill/decode asymmetry: these phases stress different system resources and are hard to co-optimize.
  • Scheduling under dependency constraints: agent steps depend on prior outputs, limiting naive batching.

Problems Solved by DualPath

  • Reduces unnecessary storage traffic by separating and coordinating workload paths.
  • Improves reuse of hot model state for prefill-heavy turn transitions.
  • Balances speculative decoding to keep throughput high without excessive wasted work.
  • Raises practical serving efficiency in realistic agentic traces rather than synthetic single-pass workloads.
Key claim from the paper: storage-aware orchestration can unlock major gains even without changing base model weights.

Results and Evidence

Performance Deltas Reported

  • Up to 1.73× higher throughput over baselines.
  • Up to 1.61× lower median end-to-end latency.
  • Up to 90.9% reduction in storage bandwidth usage.
  • About 77.3% token acceptance rate in the tested speculative setup.

Interpretation

These numbers indicate that DualPath is not just a micro-optimization; it materially shifts system-level efficiency by attacking storage traffic directly.

Other Significant Insights

  • The paper shows why optimization targets must move beyond FLOPs and include end-to-end data movement costs.
  • Speculation quality (acceptance rate) strongly influences practical gains; the system-level sweet spot matters more than maximal speculation depth.
  • Storage-aware scheduling becomes more valuable as agent depth and tool-calling frequency increase.
  • Serving architecture can deliver large wins without requiring model retraining.

Summary

DualPath is a systems contribution focused on a key pain point in agentic LLM inference: storage bandwidth pressure from repeated, fragmented execution. By coordinating hot-prefill and speculative decoding paths, it cuts read amplification and improves both throughput and latency. The reported gains suggest storage-centric design should be a first-class concern in production LLM serving.

Actionable Lessons

  • Profile storage traffic and weight movement first; do not assume compute is the top bottleneck.
  • Separate prefill-oriented and decode-oriented scheduling paths when workloads are highly turn-fragmented.
  • Tune speculative decoding using acceptance-rate telemetry, not just theoretical speedup.
  • Use agent-trace benchmarks (multi-step/tool-use) instead of only single-turn benchmarks for serving decisions.
  • Track read amplification as a primary KPI for agentic-serving architecture changes.
Practical rule: If your agent workload shows high turn counts and frequent resumes, storage-aware orchestration is likely a high-ROI optimization target.
